# Psychology privacy hotfix — migration & rollout note

**Ветка:** `security/psy-privacy-hotfix` (от `erudit/main` @ `7dab3cc`)
**Дата:** 21.07.2026
**Статус:** не смёржено, не задеплоено. Мерж и деплой — только по отдельному разрешению.

## 1. Что входит в хотфикс

Четыре дыры приватности в уже задеплоенном модуле психолога. Редизайн workflow сюда НЕ входит.

| # | Дыра | Endpoint | Было | Стало |
|---|---|---|---|---|
| 1 | Раскрытие субъект-карты | `GET /api/v1/psy/subject-card` | Гейт только по роли кабинета. Любой психолог по произвольному `parentId`/`teacherId` получал чужие кейсы, встречи и **реальные ФИО детей с классами** | psy-RLS (`caseWhereForScope` через `AND`) + филиальный гейт по субъекту + фильтр детей по филиалу + маскировка имени в `psyCode`, если нет доступного кейса по субъекту |
| 2 | Кросс-филиальное создание кейса | `POST /api/v1/psy/cases` | Нет `canAccessStudent`: кейс создавался на ученика **любого филиала** по голому `studentId` | `canAccessStudent` для `subjectType=student`; branch-scope для `parent`/`teacher`; обход только по capability |
| 3 | Доверие клиентскому баллу | `POST /api/v1/psy/screening/campaigns/[id]/submit` | `score` брался **из тела запроса** ученика и сравнивался с порогом риска — ученик сам решал, попадёт ли он в группу риска | Балл считается на сервере из `rawScores`; клиентский `score` игнорируется |
| 4 | Клинический текст администрации | `GET /api/v1/psy/safeguarding` | `reason` (обоснование красного риска) отдавался всем `COORDINATOR_ROLES`, включая завуча | `reason` выдаётся **только по явной capability** `psy_safeguarding_reason`; ни одна роль сама по себе его не открывает |

## 2. Миграция БД

**Не требуется.** Новых полей и таблиц нет, `db push` / `prisma migrate` не нужны.

Обе capability хранятся в **существующей** таблице `ModuleGrant`:

```
ModuleGrant { userId, module = 'psy_cross_branch',        canRead = true }  // межфилиальный доступ
ModuleGrant { userId, module = 'psy_safeguarding_reason', canRead = true }  // клинический reason
```

Обе по умолчанию **выключены**: строки нет → `hasPsyCapability()` возвращает `false` → работает изоляция.

### Кто видит `reason` после хотфикса

**Только пользователь с `ModuleGrant(module='psy_safeguarding_reason', canRead=true)`.** Роль сама по себе не даёт ничего:

| Роль | Доступ к очереди `/psy/safeguarding` | Поле `reason` без capability |
|---|---|---|
| `safeguarding_lead` | да | **null** |
| `zavuch` (+ `zavuch_primary/senior/academic`, `cambridge_coord`) | да | **null** |
| `super_admin` (директор) | да | **null** |
| `secretary`, прочая администрация | **нет** (401/403) | — |

Безопасная часть payload остаётся всем, у кого есть доступ к очереди: инициалы ученика, уровень риска, статус, даты, счётчик эскалаций. Тесты, беседы, заметки и заключения через этот эндпоинт не отдаются вообще.

> **Внимание при выкате:** по умолчанию ЗВР-специалист перестанет видеть обоснование, и очередь станет малоинформативной. Это осознанный fail-closed. Реальному ответственному за защиту детей грант нужно выдать **до** выката:
> ```sql
> INSERT INTO "ModuleGrant" ("id", "userId", "module", "canRead", "canWrite", "canApprove", "createdAt")
> VALUES (gen_random_uuid()::text, '<SAFEGUARDING_USER_ID>', 'psy_safeguarding_reason', true, false, false, now());
> ```

## 3. Риск отсечения действующих психологов

`getBranchScope` возвращает `closed: true`, если у пользователя `branchId = null`. После хотфикса такой психолог получит **403 на создание кейса** и на субъект-карту.

Это осознанный fail-closed: проверка не ослабляется. Но перед деплоем нужно знать масштаб.

### Обязательный preflight

Запустить на production **вручную** (Руслан), read-only, без PII:

```
scripts/psy-preflight.sql
```

Ключевая метрика — **`psy_users_without_branch`**: пользователи с ролями `psychologist`,
`senior_psychologist`, `psy_coordinator`, `specialist` и `branchId IS NULL`.

- `= 0` → деплой безопасен по этой оси.
- `> 0` → **до деплоя** либо проставить им `User.branchId`, либо выдать capability:

```sql
-- выдать межфилиальный доступ конкретному пользователю (id взять заранее)
INSERT INTO "ModuleGrant" ("id", "userId", "module", "canRead", "canWrite", "canApprove", "createdAt")
VALUES (gen_random_uuid()::text, '<USER_ID>', 'psy_cross_branch', true, false, false, now());
```

Снятие — удалить строку. Никакого редеплоя capability не требует.

Также посмотреть `students_without_branch`: ученики без филиала недоступны branch-скоупленным ролям.

## 4. Порядок выката

1. Прогнать `scripts/psy-preflight.sql` на проде, зафиксировать цифры.
2. Закрыть `psy_users_without_branch` (branchId или capability `psy_cross_branch`).
3. Выдать `psy_safeguarding_reason` ответственному за защиту детей — иначе очередь останется без обоснований.
4. Прогнать `e2e/psy-privacy-hotfix.spec.ts` на локальной базе с сидом при поднятом `npm run dev`.
5. Решение о мерже — отдельно.
6. После деплоя — смоук: психолог открывает кейс своему ученику, отправка скрининга учеником, ЗВР открывает safeguarding и видит обоснование, завуч — не видит.

## 5. Откат

Ревёрт коммита ветки. Миграций нет, данные не менялись, обратной несовместимости нет.
Строки `ModuleGrant` с `module='psy_cross_branch'` и `'psy_safeguarding_reason'` после отката просто игнорируются.

## 6. Изменение поведения, которое заметят пользователи

- Ученик, отправляющий скрининг с некорректными ответами (не число, вне шкалы шаблона, пустая анкета), получит `400` вместо молчаливого сохранения. Границы берутся из `template.schema` (`scaleMin`/`scaleMax`, фолбэк 0..10).
- Обоснование красного риска в safeguarding-очереди перестанут видеть **все** — завуч, директор (`super_admin`) и сам `safeguarding_lead`, — пока им не выдана capability `psy_safeguarding_reason`. Факт риска, инициалы, уровень и статус остаются.
- Психолог без доступного кейса по родителю увидит в субъект-карте `psyCode` вместо ФИО ребёнка.

## 7. Что НЕ входит и остаётся открытым

- **Права `psy_coordinator`** на правку/закрытие/переназначение чужих кейсов. Голосовое психолога подтверждает только просмотр, но код уже сейчас разрешает координатору `transfer` и управление коллабораторами. Не трогали намеренно — `POLICY_OPEN`, нужен ответ Эмира.
- **Межфилиальный scope старшего психолога** — `POLICY_OPEN`; сейчас закрыт по умолчанию, открывается capability.
- **Audit trail** psy-действий отсутствует в системе целиком — отдельная задача.
- **Слабое место `subjectBranchIds`:** филиал родителя выводится через филиалы его детей; родитель с детьми в двух филиалах попадёт в оба скоупа. Для MVP приемлемо, в бэклог.
